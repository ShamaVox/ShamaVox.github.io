Why teams call me

Every AI project I have watched fail, failed the same way.

The demo worked. Everyone was happy. Then real data showed up, or a real customer did, and the thing that looked finished turned out to be about 40% of the work.

The missing 60% is almost always the same. Nothing checks whether the model was right. The model decides when it is done. Retries run until the bill arrives. And when something breaks, nobody can reconstruct what happened.

I have built that layer four times. I would rather help you build it before the rewrite than after.

01  /  One week

Architecture audit

Fixed scope. Written findings.

You have a prototype or a live system and you are about to scale it, ship it, or raise on it.

  • I read the code and the architecture, not the deck
  • Where the model is making decisions your code should be making
  • Retry behaviour, token spend, and what happens when the loop does not stop
  • API design, data handling, and the security holes prototypes usually carry
  • Whether you can reconstruct, afterward, why it did what it did
  • What your team is missing and who to hire for it

You getA written list of what will break, ranked by what it costs when it does. Sometimes the answer is that it is fine. I will say that too.

02  /  Two to six weeks, full time

Agentic build sprint

Hands on the keyboard, in your repo.

You need the thing built, or you have a model that works most of the time and most of the time is not a product you can charge for.

  • Multi-agent workflows and autonomous pipelines, from nothing or from your prototype
  • RAG, retrieval, and ingestion architecture that survives real documents
  • Typed schemas and deterministic validation at the boundary
  • Bounded retry, so a bad output fails once instead of forever
  • Confidence scoring, evals, and a threshold you can actually tune
  • Tracing, so you can see which node burned the tokens

You getWorking code your engineers can extend. This is the work that took my last document pipeline to 96% in production.

Selected work

Same pattern in all of them.

Something generates, something else checks it, and there is a defined path for when the check fails. That is what separates a system a company will run from one that only demos well.

AI-native retail platform

Multi-agent document intelligence

Context
Retailers were losing thousands of hours to manual data entry from vendor documents. Usually right is not a standard you can run a business on.
Architecture
Multi-agent extraction turning messy vendor documents into typed, validated records. Confidence scoring on every field, bounded retry instead of open-ended loops, and escalation to a human when the score drops. When it fails it fails into a known state rather than a plausible wrong one.
Result
96% extraction accuracy in production with live paying merchants. Built a custom onboarding path around a platform certification bottleneck so sales did not stall.
Stack
Claude API, multi-agent orchestration, confidence scoring, human review tooling, commerce platform integration.

Result

Edge computer vision, real-time geospatial tracking, attribution pipeline, cloud infrastructure engineered for sublinear cost scaling.
SignAI

Real-time multimodal translation

Context
Sign language interpretation is a latency problem dressed up as a translation problem. An interpreter that lags is not an interpreter.
Architecture
Bidirectional real-time ASL and English interpretation using gesture and pose modeling, low-latency inference, and edge optimization on commodity hardware.
Result
Delivered a working real-time bidirectional prototype on commodity hardware and ran technical validation demos for Microsoft. Handed the platform off as a Deaf-led initiative. Accessibility work belongs with the community it serves.
Stack
Computer vision, gesture and pose modeling, low-latency inference, bidirectional translation pipeline.
BrainEx · Side project

Confidential support for students in India

Context
Students in India face significant mental health pressure and very little confidential support. The barrier is rarely willingness to seek help. It is the fear that seeking it will be visible to family, faculty, or peers.
Architecture
A privacy-first data architecture where confidentiality is a system property rather than a policy document. Strict access controls, data minimisation, and audit logging on every sensitive read.
Result
A support workflow students could actually use, with the privacy guarantees enforced in the architecture rather than promised on a terms page.
